## Overview



`ot2\_module` is a MADSci node module for Opentrons liquid handlers (both OT-2 and Flex). It exposes a FastAPI REST server that integrates with the MADSci framework, and communicates with the physical robot over its built-in HTTP API (port 31950).



See `MADSci/CLAUDE.md` (or `MADSci/AGENTS.md`) for MADSci framework architecture, node module patterns, and conventions this module follows.

\## Architecture



\### Entry Point

`src/ot2\_rest\_node.py` â€” `OT2Node(RestNode)` is the MADSci node implementation. It:

\- Extends `RestNodeConfig` with `ot2\_ip: Optional\[str]`

\- Connects to the robot via `OT2\_Driver` on startup using `self.config.ot2\_ip`

\- Registers one action: `run\_protocol` (decorated with `@action`)

\- Implements `startup\_handler`, `shutdown\_handler`, `state\_handler`, and control methods (`pause`, `resume`, `cancel`, `safety\_stop`, `reset`)

\- Uses `self.node\_info.node\_name` / `self.node\_info.node\_id` for identity (not a definition file)



\### Node Identity (settings-based pattern)

Node identity (`node\_name`, `node\_id`, etc.) is configured via `settings.yaml` or env vars â€” there are no separate definition files. The `NodeInfo` object is built automatically from the config by `NodeInfo.from\_config()` in the MADSci base class.



\### OT2 HTTP Driver

`src/ot2\_interface/ot2\_driver\_http.py` â€” `OT2\_Driver` wraps the Opentrons REST API:

\- `transfer(protocol\_path)` â†’ uploads a `.py` protocol file, creates a run, returns `(protocol\_id, run\_id)`

\- `execute(run\_id)` â†’ sends a `play` action, polls until run completes (SUCCEEDED/FAILED/STOPPED)

\- `pause(run\_id)`, `resume(run\_id)`, `cancel(run\_id)` â†’ lifecycle control

\- `get\_run\_log(run\_id)` â†’ fetches run data + commands (up to 1000)

\- `compile\_protocol(config\_path, ...)` â†’ delegates YAMLâ†’Python compilation to Protopiler (skips if `.py` already)



`OT2\_Config` (in `src/ot2\_interface/config.py`): `ip`, `port` (default `31950`), `model`.



\### Protopiler

`src/ot2\_interface/protopiler/` â€” Compiles YAML protocol configs into Opentrons Python protocol files using string templates.



\*\*Config structure\*\* (`ProtocolConfig` in `protopiler/config.py`):

\- `equipment`: list of `Labware` (deck slot + name + optional alias) and `Pipette` (name + mount)

\- `commands`: list of typed command objects (`Transfer`, `Multi\_Transfer`, `Mix`, `Move\_Labware`, `Temperature\_Set`, `Deactivate`, `Replace\_Tip`, `Clear\_Pipette`, `Move\_Pipette`, `Ninetysix\_Transfer`)

\- `metadata`: protocol name, author, description, apiLevel

\- `resources`: optional list of external files (xlsx) for data-driven transfers



\*\*Transfer command\*\* fields include `source`, `destination` (e.g. `"source:A1"` using labware aliases), `volume`, `aspirate\_clearance`, `dispense\_clearance`, `mix\_cycles`, `blow\_out`, `drop\_tip`, `return\_tip`. List fields must all be the same length.



Templates live in `protopiler/protocol\_templates/\*.template`. Example configs are in `protopiler/test\_configs/`.



\### Parameter Substitution in `run\_protocol`

The `run\_protocol` action accepts a `parameters` dict. Before execution, occurrences of `$key` in the protocol `.py` file are replaced with the corresponding values. This is a simple string substitution (not Jinja2).



\### Protocol Execution Flow

1\. MADSci sends `run\_protocol` action with a protocol `.py` file and optional parameters

2\. `run\_protocol` substitutes parameters into the file, then calls `execute()`

3\. `execute()` calls `ot2\_interface.transfer()` (upload + create run) then `ot2\_interface.execute()` (play + poll)

4\. On success, `get\_run\_log()` is called and optionally used to update MADSci resources



\### Docker Deployment

\- Base image: `ghcr.io/ad-sdl/madsci`

\- Published image: `ghcr.io/ad-sdl/ot2\_module`

\- Install: `uv pip install --python ${MADSCI\_VENV}/bin/python -e /home/madsci/ot2\_module`

\- Uses `network\_mode: host` (required to reach OT-2 on the local network)

\- Volumes: `settings.yaml` and `.madsci/` are bind-mounted (no definition files)

\- `USER\_ID` / `GROUP\_ID` build args control file ownership

\- Ruff is configured with `target-version = "py310"` and pydocstring requirements (`D1xx`)
